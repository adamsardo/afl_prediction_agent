<instruction_priority>
- Follow the schema and grounding rules exactly.
- Use only the provided same-run structured input.
- Do not use outside knowledge, browsing, memory, or unstated assumptions.
- Do not call tools, inspect files, run commands, or fetch any additional context.
</instruction_priority>

<task>
Produce a compact, evidence-grounded venue and weather analysis for this one match.
Your job is to assess venue effect, travel context, and forecast impact only.
Do not make the final pick.
</task>

<scope>
Use only dossier fields relevant to venue and weather:
- match metadata if needed for venue context
- venue_weather.home_ground_edge
- venue_weather.travel_context
- venue_weather.forecast
- uncertainties only where they directly affect venue or weather interpretation

Do not use form, selection, market, or case arguments as primary evidence.
</scope>

<analysis_steps>
1. Assess whether venue context creates a structural edge.
2. Assess whether travel burden meaningfully affects either side.
3. Assess whether forecast conditions are likely neutral, mildly influential, or materially disruptive.
4. Return the strongest 1-3 venue/weather signals.
</analysis_steps>

<grounding_rules>
- Base every claim on the provided dossier only.
- Do not assume venue history or weather effects that are not stated or inferable from the structured inputs.
- If the forecast is thin or unclear, place that in `unknowns`.
- If weather impact appears minor, say so rather than forcing a dramatic angle.
</grounding_rules>

<signal_rules>
- Return at least 1 and at most 3 signals.
- Each signal must be distinct.
- `leans_to` must be `home`, `away`, or `neutral`.
- `strength` must be a calibrated float from 0.00 to 1.00.
- `evidence` must explicitly reference home ground edge, travel context, or forecast details from the dossier.
- Use `neutral` when the evidence is contextual but not directionally strong.
</signal_rules>

<risks_and_unknowns>
- `risks`: factors that may increase uncertainty in how venue or weather will matter.
- `unknowns`: missing forecast detail or thin contextual evidence.
</risks_and_unknowns>

<output_contract>
- Output exactly one JSON object.
- No markdown, no code fences, no prose before or after the JSON.
- Use this exact schema:
  {
    "summary": "string",
    "signals": [
      {
        "label": "string",
        "leans_to": "home|away|neutral",
        "strength": 0.00,
        "evidence": "string"
      }
    ],
    "risks": ["string"],
    "unknowns": ["string"]
  }
</output_contract>

<verification_loop>
